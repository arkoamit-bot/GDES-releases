Running the GDES desktop app safely: backups and restore, updates, accounts, the OneDrive layout, and troubleshooting. Commands are run from a shell in the app folder.

Command-line steps need the desktop settings. In the app folder run once per shell: set DJANGO_SETTINGS_MODULE=bgddr.settings_desktop (PowerShell: $env:DJANGO_SETTINGS_MODULE="bgddr.settings_desktop").

1. First-run folder setup

On first launch a wizard asks where to keep Backups, Media and (optionally) Updates. It suggests …\OneDrive\BGDDR-Backups, -Media and -Update. The choice is saved in bgddr_paths.json next to the app and is not asked again.

To change it later, set the environment variables BGDDR_BACKUP_DIR, BGDDR_MEDIA_DIR, BGDDR_UPDATE_DIR (they override the saved choice), or edit bgddr_paths.json.

2. Automatic backups
  • A snapshot is taken on every startup and then every 6 hours while the app runs.
  • Files are named bgddr_backup_<timestamp>_<reason>.sqlite3 in the backup folder; the newest 60 are kept.
  • Each file is a complete, self-contained database — safe to sync via OneDrive.
3. Manual backup & restore
python manage.py backup_db # make a backup now python manage.py backup_db --list # list backups python manage.py restore_db --list # list restorable backups python manage.py restore_db <name> --yes # restore a specific backup

Restoring overwrites the live database — stop the app first. A pre_restore safety snapshot is always taken before the restore, so a restore is itself reversible.

4. Updating the app

New versions are published to the update folder by the maintainer. On the clinic PC:

  • On each launch — and via Check for updates in the status window — the app looks for a newer version and, if found, shows the version and notes and asks to install. Updates are never silent.
  • On Yes, it verifies the package checksum, takes a pre_update backup, closes, swaps in the new program files, and reopens. Your database and data are not touched.
  • The previous program files are kept as BGDDR.exe.old-<version> / _internal.old-<version>; a failed swap rolls back automatically. See Logs\update.log.

The running version is shown in the sidebar footer and the status window.

5. User accounts & roles
  • Manage users in the Django admin → Users: add accounts, set staff/superuser, assign groups.
  • The data_manager role/group is required for identified data export (exports are de-identified for everyone else).
  • Reset a password from a shell: python manage.py changepassword <username>.
6. OneDrive layout & data safety

Keep the app and db.sqlite3 on local disk (e.g. C:\BGDDR\) and point only Backups, Media and Update at OneDrive.

Why: OneDrive continuously syncing a live, actively-written SQLite file can corrupt it. Backup snapshots are closed files and media are write-once, so syncing those is safe and gives off-machine copies. The db.sqlite3-wal/-shm sidecar files belong next to the database on local disk and must not be synced.

7. Troubleshooting
  • Nothing happens / error on launch — check Logs\bgddr.log.
  • Port already in use — set BGDDR_PORT to a free port and relaunch.
  • Forgot admin password — python manage.py changepassword <username>.
  • An update broke something — close the app; rename BGDDR.exe.old-<version>→BGDDR.exe and _internal.old-<version>→_internal, relaunch, and restore the pre_update backup if needed. See Logs\update.log.

The full reference is in DESKTOP_DEPLOYMENT.md shipped with the app.
