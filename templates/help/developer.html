1. Project layout & settings

A standard multi-app Django project (patients, encounters, labs, prescriptions, clinical_reasoning, knowledge, followup, clinic UI, …) on SQLite. Three settings modules:

  • bgddr/settings.py — base/dev (DEBUG on, insecure key). Used by manage.py by default.
  • bgddr/settings_desktop.py — the packaged app: DEBUG off, persistent SECRET_KEY, localhost only, data under BGDDR_DATA_DIR. This is what the launcher uses.
  • bgddr/settings_prod.py — future multi-user/PostgreSQL deployment.

App version lives in bgddr/version.py (__version__) and is surfaced in the UI via a context processor.

2. Running from source
python manage.py runserver # dev server (base settings, DEBUG on) python desktop\launcher.py # desktop stack (Waitress + settings_desktop) Start-BGDDR.bat # double-click launcher (uses the launcher)

For anything user-facing, prefer launcher.py / Start-BGDDR.bat: it uses the hardened desktop settings, whereas bare runserver uses the dev settings.

3. Building the .exe
.\desktop\build_exe.ps1 # installs build deps, rebuilds CSS, runs PyInstaller, self-checks # manual equivalent: pyinstaller desktop\BGDDR.spec --noconfirm dist\BGDDR\BGDDR.exe --check # headless smoke test

Output is dist\BGDDR\ (BGDDR.exe + _internal\) — the deployable package. The --check run migrates, seeds, collects static, binds the port and verifies the SPSS writer.

When adding a Django app, also add it to LOCAL_APPS in desktop/BGDDR.spec. Django imports apps dynamically, so PyInstaller won’t find one that’s missing from that list — the build can “succeed” and then crash or drop templates at runtime.

4. Publishing an update
  1. Bump __version__ in bgddr/version.py (dotted numeric, e.g. 6.6.0).
  2. Build: .\desktop\build_exe.ps1.
  3. Zip dist\BGDDR as BGDDR-<version>.zip (must contain BGDDR.exe + _internal\, top-level or one folder deep).
  4. Checksum: Get-FileHash BGDDR-<version>.zip -Algorithm SHA256.
  5. Copy the zip into the OneDrive update folder and write latest.json:
{ "version": "6.6.0", "file": "BGDDR-6.6.0.zip", "sha256": "<hash from step 4>", "notes": "What changed in this release." }

Once OneDrive syncs to the clinic PC, the next launch offers the update. The app refuses a package whose SHA-256 does not match — only a trusted maintainer should have write access to the update folder (it runs code on the clinical machine). Logic lives in bgddr/updater.py; the swap helper preserves data and keeps *.old-<version> for rollback.

Before you build: if this release changes a workflow, update the in-app Help guides (templates/help/user.html, admin.html, developer.html) in the same commit so the docs ship in sync with the version they describe.

5. CSS & static assets
npm run build:css # static_src/input.css -> static/css/app.css (Tailwind) python manage.py collectstatic --noinput
  • Templates load app.css via the {% templatetag openblock %} static_v 'css/app.css' {% templatetag closeblock %} tag (in clinic/templatetags/asset_tags.py), which appends ?v=<mtime> so browsers pick up rebuilds automatically — no hard refresh.
  • Gotcha: several component styles (.tab-list, .card, …) are defined in both app.css and an inline <style> block in templates/base.html. The inline block loads later, so it wins — edit component styles there, not only in input.css.
6. Config & environment variables
VariablePurpose
BGDDR_DATA_DIRapp data root (db, Exports, Logs) — keep on local disk
BGDDR_BACKUP_DIRbackup-snapshot folder (OneDrive-syncable)
BGDDR_MEDIA_DIRuploads/PDF folder (OneDrive-syncable)
BGDDR_UPDATE_DIRpublished-updates folder; blank disables in-app updates
BGDDR_PORTlocal server port (default 8000)
BGDDR_MAX_BACKUPS / BGDDR_BACKUP_INTERVAL_HOURSretention / interval
DJANGO_DB_ENGINE + POSTGRES_*switch to PostgreSQL (future)

The launcher’s first-run wizard writes backup/media/update choices to bgddr_paths.json and exports them as the env vars above before Django loads; an explicit env var always wins.

7. Database, migrations & WAL
  • SQLite in WAL journal mode (set via OPTIONS["init_command"]) for crash resilience — creates db.sqlite3-wal/-shm sidecars that stay on local disk.
  • Migrations and reference-data seeds (seed_roles, seed_labs, seed_drugs, seed_studies) run automatically on launch; first run is guarded by a .initialized marker.
  • The schema is ORM-only and PostgreSQL-compatible (no SQLite-specific features, JSONField→jsonb).
8. Future: PostgreSQL

No code changes needed — the backend is env-selected. Dump ORM-native data, set DJANGO_DB_ENGINE=postgres + POSTGRES_*, then migrate and loaddata. Use settings_prod.py behind a real WSGI server for a networked, multi-user deployment. Full steps are in DESKTOP_DEPLOYMENT.md §6.
